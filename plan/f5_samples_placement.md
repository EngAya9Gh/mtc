# f5 — SAMPLES PLACEMENT (وضع العينات في الحاوية) — توثيق شامل ومفصّل (نسخة مطوّرة ومبسطة للمطور)
# المصدر الأصلي: TaskTypeFragment ← ListTaskFragment → FreezerOutBagsScanFragment
# Base URL: https://gosample.com/api/

---

## نظرة عامة

**SAMPLES PLACEMENT** هو الخيار الثاني في شاشة اختيار نوع المهمة (`TaskTypeSelectorScreen`).
الهدف منه: وضع الأكياس التي تم جمعها مسبقاً من العميل (والتي أصبحت حالتها `COLLECTED`) داخل حاويات التبريد المناسبة الموجودة في سيارة السائق.

**المنطق الأساسي للتدفق:**
1. يختار السائق مهمة معينة من قائمة المهام المفلترة على الحالة `COLLECTED`.
2. تفتح الكاميرا في البداية لمسح **الحاوية/الثلاجة** الخاصة بالسيارة لدرجة الحرارة المطلوبة.
3. بعد تأكيد مسح الحاوية، يبدأ السائق بمسح **أكياس العينات البيولوجية** التابعة لنفس درجة الحرارة واحداً تلو الآخر حتى تكتمل.
4. يضغط السائق على زر حفظ الأكياس الخاصة بهذه الثلاجة.
5. يتكرر هذا التدفق (مسح حاوية جديدة ثم الأكياس الخاصة بها) إذا كان هناك أنواع حرارة متعددة مطلوبة للمهمة (مثل ROOM ثم REFRIGERATE ثم FROZEN).
6. بمجرد مسح وتسكين جميع الأكياس، يظهر زر إنهاء المهمة وإغلاق جميع الحاويات.

## Screen 1: TaskListScreen — شاشة قائمة المهام
### الكلاس الأصلي: `ListTaskFragment.java`

### 1. التدفق والوصول (Navigation Flow)
- **كيف تصل إليها:** من شاشة اختيار نوع المهمة (`TaskTypeSelectorScreen`) عند الضغط على زر **SAMPLES PLACEMENT / وضع العينات في الحاوية**.
- **إلى أين تأخذك:** عند الضغط على أي بطاقة مهمة في القائمة ← يتم الانتقال إلى شاشة مسح الحاوية والأكياس (`FreezerOutBagsScreen`).

### 2. عناصر التصميم ووصفها الوظيفي (UI Layout Elements)
- **شريط التنقل العلوي (AppBar):** يحتوي على زر العودة الخلفي (يتم تدويره 180 درجة للغة العربية) وعنوان الشاشة **"وضع العينات في الحاوية"** باللغة الحالية.
- **زر تأكيد الكل (Accept All Button):** يظهر في الأعلى فقط إذا كان هناك أي مهمة في القائمة لم يقم السائق بتأكيدها مسبقاً (حيث تكون قيمة `isConfirmedByDriver == false`). عند الضغط عليه، يؤكد السائق استلامه لكافة المهام دفعة واحدة.
- **مكون السحب للتحديث (Swipe-to-Refresh):** يسمح للسائق بسحب الشاشة للأسفل لإعادة تحميل قائمة المهام وتحديث حالتها من الخادم.
- **قائمة المهام (RecyclerView):** تعرض بطاقات المهام المتاحة. تحتوي كل بطاقة على:
  - اسم العميل (بالعربية أو الإنجليزية حسب لغة التطبيق).
  - حالة المهمة الحالية (COLLECTED).
  - تاريخ ووقت المهمة.
  - اسم موقع البداية واسم موقع النهاية (من ← إلى).

### 3. المنطق البرمجي والشروط (Business Logic)
```
عند فتح الشاشة (onResume):
├── تحقق من وجود بيانات السائق (UserInfo.loginInfo != null)
│   ├── إذا صحيحة ← استدعِ API جلب قائمة المهام
│   └── إذا فارغة ← أظهر خطأ
├── تحقق من لغة التطبيق الحالية:
│   └── إذا كانت العربية (ar) ← اعكس اتجاه سهم الرجوع 180 درجة
└── عيّن عنوان الشاشة = "SAMPLES PLACEMENT" / "وضع العينات في الحاوية"

زر "تأكيد الكل" (Accept All):
├── يظهر فقط في حال وجود مهام معلقة لم يؤكدها السائق بعد
└── عند الضغط عليه ← استدعِ API تأكيد المهام (confirmTaskByDriver)
    ├── نجاح العملية ← أعد جلب القائمة لتحديث الحالات
    └── فشل العملية ← أظهر رسالة خطأ (Toast)

عند الضغط على بطاقة المهمة:
├── احفظ المهمة المحددة في الحالات المشتركة للتطبيق (UserInfo.selectedTask = task)
└── انتقل فوراً إلى شاشة مسح الحاوية والأكياس (FreezerOutBagsScreen)
```

## Screen 2: FreezerOutBagsScreen — شاشة مسح الحاوية والأكياس
### الكلاس الأصلي: `FreezerOutBagsScanFragment.java`

هذه هي الشاشة المحورية في هذا التدفق. تعتمد على **كاميرا واحدة** وتصميم ديناميكي ينقسم إلى **قسمين متبادلين** لمنع تشتيت السائق.

### 1. عناصر الواجهة ووصفها الوظيفي (UI UX Description)

#### القسم الأول: قسم مسح باركود الحاوية (Container Scan Section)
يظهر هذا القسم أولاً عند الدخول إلى الشاشة، وهدفه التأكد من أن السائق يضع العينات في الثلاجة الصحيحة داخل سيارته.
- **إطار الكاميرا النشط (Camera View):** يفتح الكاميرا لمسح باركود الحاوية الملصق على الثلاجة بالسيارة.
- **حقل عرض باركود الحاوية (Container Barcode Text):** نص يعرض الباركود الذي تم مسحه حالياً لتنبيه السائق.
- **أيقونة تأكيد مسح الحاوية (Checkmark Icon):** أيقونة خضراء تظهر بجانب الباركود الممسوح عند نجاح عملية المسح والتحقق منها.
- **زر حفظ الحاوية الممسوحة (SAVE CONTAINER Button):** يظهر فقط بعد إتمام مسح باركود حاوية بنجاح. عند الضغط عليه، يتم إخفاء هذا القسم وفتح قسم مسح الأكياس تلقائياً.

#### القسم الثاني: قسم مسح أكياس العينات (Bag Scan Section)
يظهر هذا القسم مباشرة بعد قيام السائق بالضغط على زر "حفظ الحاوية".
- **شريط فئات الحرارة (Temperature Header):** ثلاثة أزرار تمثل درجات الحرارة للأكياس المطلوبة في المهمة:
  - **ROOM** (حرارة الغرفة).
  - **REF** (التبريد).
  - **FRZ** (التجميد).
  *ملاحظة للمطور:* هذه الأزرار تفاعلية للعرض فقط (معطلة الضغط)، حيث يقوم التطبيق بتحديد درجة الحرارة النشطة تلقائياً ويبدأ دائماً بالـ ROOM، ثم الـ REF، ثم الـ FRZ. يتم تلوين الزر النشط باللون الأزرق والرمادي للبقية.
- **إطار الكاميرا النشط (Camera View):** يتحول عمل الكاميرا هنا لمسح باركودات الأكياس المغلقة الموجودة في يد السائق.
- **قائمة الأكياس المطلوب مسحها (Remaining Bags RecyclerView):** تعرض قائمة بالأكياس المتبقية التي لم يتم مسحها بعد للنوع الحالي النشط. يقل عدد الأكياس في القائمة كلما مسح السائق كيساً بنجاح.
- **زر حفظ الأكياس الممسوحة (SAVE BAGS Button):** يظهر هذا الزر في الأسفل فقط عندما يمسح السائق **جميع الأكياس** التابعة للنوع الحالي وتصبح القائمة فارغة. عند الضغط عليه يظهر تنبيه لتأكيد الحفظ على الخادم.
- **زر إغلاق الحاويات وإنهاء المهمة (CLOSE FREEZERS Button):** يظهر هذا الزر فقط بعد حفظ آخر نوع حرارة متبقي في المهمة بالكامل وتأكيده بنجاح على الخادم.

---

### 2. المنطق البرمجي التفصيلي للشاشة (Business Logic)

#### أ. تدفق مسح باركود الحاوية:
* **توضيح هام جداً للبرمجة:** عند مراجعة الكود الفعلي للتطبيق الأندرويد، تبين أن التطبيق **لا يقوم بأي عملية مطابقة أو تحقق محلي (Local Validation)** على باركود الحاوية الممسوح ولا يقارنه محلياً ببيانات السيارة.
* المنطق الفعلي المطبق هو:
  1. بمجرد أن يمسح السائق أي باركود أثناء ظهور قسم مسح الحاوية، يتم استقبال الكود مباشرة.
  2. يتم حفظ الكود الممسوح في متغير `selectedContainerQrCode`.
  3. يتم عرض النص الممسوح مباشرة على الشاشة وتظهر أيقونة التأكيد الخضراء وزر **حفظ الحاوية**.
  4. عند الضغط على زر حفظ الحاوية، ينتقل السائق فوراً لمسح الأكياس.
  5. يتم إرسال كود الحاوية الممسوح للخادم لاحقاً عبر API الحفظ (`samples/container/bags/add`)، **والخادم هو المسؤول بالكامل عن التحقق من صحة الحاوية ومطابقتها**. في حال كانت الحاوية غير صحيحة، يرجع الخادم استجابة فشل (`status: false`)، وهنا يقوم التطبيق محلياً بإعادة تصفير الأكياس وإرجاع السائق لمسح الحاوية مجدداً.

#### ب. تدفق مسح أكياس العينات:
1. بعد حفظ الحاوية، يفتح قسم مسح الأكياس، ويتم تلوين زر فئة الحرارة المطابقة للحاوية الممسوحة باللون الأزرق.
2. يمسح السائق باركود الكيس البيولوجي:
   - يبحث التطبيق داخل قائمة الأكياس المتبقية للنوع النشط الحالي: هل الباركود الممسوح ينتمي لهذه القائمة؟
     - **نعم:** يتم إزالة الكيس الممسوح من القائمة وتحديث القائمة في الواجهة مباشرة.
     - **لا:** يتم تجاهل المسح وإصدار تنبيه صوتي/اهتزاز خفيف لعدم تطابق الكيس مع فئة الحرارة الحالية.
3. عند مسح آخر كيس في القائمة النشطة، تفرغ القائمة ويظهر زر **"SAVE BAGS / حفظ الأكياس"**.

#### ج. لوجيك إرسال الأكياس إلى الخادم (حفظ فئة الحرارة):
```
عند الضغط على زر SAVE BAGS:
├── أظهر حوار تأكيد للسائق: "هل أنت متأكد من حفظ أكياس الحاوية؟"
└── عند تأكيد السائق (Yes) ← استدعِ API الإرسال المجمع (samples/container/bags/add)
    ├── في حال نجاح الاستجابة من الخادم (status == true):
    │   ├── أخفِ زر SAVE BAGS
    │   ├── حدّث شريط فئات الحرارة لإخفاء الفئة التي اكتملت تماماً
    │   └── تحقق: هل تم مسح وحفظ جميع أكياس المهمة لكافة فئات الحرارة?
    │       ├── نعم (اكتملت المهمة بالكامل) ← أخفِ الكاميرا وأظهر زر "CLOSE FREEZERS / إغلاق الحاويات"
    │       └── لا (تبقت فئات حرارة أخرى) ← قم بإعادة السائق تلقائياً إلى "قسم مسح الحاوية" 
    │               مع تحديد فئة الحرارة التالية المطلوبة (مثلاً الانتقال من ROOM إلى REF) 
    │               ليقوم بمسح ثلاجة التبريد الجديدة بالسيارة.
    └── في حال فشل الاستجابة (status == false) أو انقطاع الشبكة:
        ├── أظهر رسالة الخطأ الواردة من الخادم
        └── استدعِ دالة إعادة التهيئة (reloadTheBags):
            ├── أعد تعبئة قائمة الأكياس للنوع الحالي بالكامل من النسخة الاحتياطية الأصلية
            └── أعد عرض "قسم مسح الحاوية" ليعيد السائق مسح الحاوية والأكياس للنوع الحالي من البداية
```

#### د. لوجيك إنهاء وإغلاق المهمة بالكامل:
```
عند الضغط على زر CLOSE FREEZERS:
├── أظهر حوار تأكيد غير قابل للإغلاق العشوائي (setCanceledOnTouchOutside = false)
└── عند تأكيد السائق (Yes) ← استدعِ API إغلاق المهمة (task/freezer)
    ├── نجاح العملية ← انتقل فوراً إلى شاشة النجاح (TaskStatusScreen)
    └── فشل العملية ← أظهر رسالة خطأ للسائق مع إبقائه في نفس الشاشة لإعادة المحاولة
```

## مدخلات ومخرجات كافة الـ APIs بالتفصيل (API Specifications)

### 1. API: جلب قائمة المهام (driver/tasks)
* **الغرض:** جلب المهام النشطة للسائق المحددة بحالة معينة.
* **رابط الاتصال (Endpoint):** `POST` لـ `driver/tasks`
* **نوع المحتوى (Content-Type):** `application/json`
* **المدخلات (Request Body):**
  ```json
  {
    "driver_id": 23,        // معرف السائق المسجل (Integer)
    "status": "COLLECTED"   // حالة المهمة المطلوب فلترتها (String)
  }
  ```
* **المخرجات الناجحة (Response Body):**
  ```json
  {
    "status": true,
    "message": "Tasks fetched successfully",
    "data": [
      {
        "id": 45,
        "from_location": 10,
        "to_location": 20,
        "from_location_name": "عيادة السلام",
        "to_location_name": "المختبر المركزي",
        "client_arabic_name": "مختبر البرج",
        "client_english_name": "Al Borg Lab",
        "dateString": "2026-05-18",
        "timeString": "15:00:00",
        "confirmed_received_by_driver": 1, // 1 يعني تم التأكيد، 0 يعني معلق
        "status": "COLLECTED",
        "numberOfBags": 3,
        "rtCount": 1,        // عدد أكياس الغرفة (ROOM)
        "refCount": 1,       // عدد أكياس التبريد (REFRIGERATE)
        "frzCount": 1        // عدد أكياس التجميد (FROZEN)
      }
    ]
  }
  ```

### 2. API: تأكيد استلام المهمة (driver/tasks/confirm)
* **الغرض:** تأكيد السائق أنه سيبدأ العمل على المهام المسندة إليه.
* **رابط الاتصال (Endpoint):** `POST` لـ `driver/tasks/confirm`
* **نوع المحتوى (Content-Type):** `application/x-www-form-urlencoded`
* **المدخلات (Form Fields):**
  * `task_ids[]` : `[45, 46]` (مصفوفة من الأرقام الصحيحة تمثل معرفات المهام التي سيتم تأكيدها دفعة واحدة).
* **المخرجات الناجحة (Response Body):**
  ```json
  {
    "status": true,
    "message": "Tasks confirmed successfully"
  }
  ```

### 3. API: جلب أكياس المهمة (task/bags/get)
* **الغرض:** جلب قائمة الأكياس والباركودات التابعة لمهمة معينة تم جمعها في المرحلة الأولى.
* **رابط الاتصال (Endpoint):** `POST` لـ `task/bags/get`
* **نوع المحتوى (Content-Type):** `application/x-www-form-urlencoded`
* **المدخلات (Form Fields):**
  * `task_id` : `45` (معرف المهمة الحالية).
* **المخرجات الناجحة (Response Body):**
  ```json
  {
    "status": true,
    "message": "Bags list retrieved successfully",
    "data": {
      "bags": [
        {
          "bag_code": "BAG-ROOM-001",
          "barcode_id": null,
          "sample_type": "NORMAL",
          "temperature_type": "ROOM" // درجة حرارة الكيس: ROOM, REFRIGERATE, FROZEN
        },
        {
          "bag_code": "BAG-REF-002",
          "barcode_id": null,
          "sample_type": "NORMAL",
          "temperature_type": "REFRIGERATE"
        }
      ]
    }
  }
  ```

### 4. API: تسكين وحفظ الأكياس في الحاوية (samples/container/bags/add)
* **الغرض:** إرسال ومطابقة قائمة الأكياس مع باركود الحاوية الذي تم مسحه.
* **رابط الاتصال (Endpoint):** `POST` لـ `samples/container/bags/add`
* **نوع المحتوى (Content-Type):** `application/json`
* **المدخلات (Request Body - JSONArray):**
  ```json
  [
    {
      "task_id": 45,
      "container_id": "CONT-ROOM-12",  // باركود الحاوية النصي الممسوح من السيارة
      "bag_code": "BAG-ROOM-001"       // باركود الكيس الممسوح
    }
  ]
  ```
  *(ملاحظة: يتم إرسال طلب مجمع لكل فئة حرارة بشكل منفصل فور اكتمال مسح أكياسها).*
* **المخرجات الناجحة (Response Body):**
  ```json
  {
    "status": true,
    "message": "Samples added to container successfully"
  }
  ```

### 5. API: إغلاق الحاويات للمهمة (task/freezer)
* **الغرض:** إغلاق الحاويات وإنهاء تدفق تسكين العينات نهائياً للمهمة لتتحول حالتها لـ `IN_FREEZER`.
* **رابط الاتصال (Endpoint):** `POST` لـ `task/freezer`
* **نوع المحتوى (Content-Type):** `application/json`
* **المدخلات (Request Body):**
  ```json
  {
    "task_id": 45
  }
  ```
* **المخرجات الناجحة (Response Body):**
  ```json
  {
    "status": true,
    "message": "Task freezer closed successfully"
  }
  ```

## ملخص الحالات والمتغيرات المشتركة المطلوبة للمطور (State Management)

يجب على المطور الاحتفاظ بهذه المتغيرات أثناء التنقل بين الأقسام لضمان سلاسة العمل:

```dart
// متغيرات المهمة الحالية
String selectedTaskType = "COLLECTED"; // يتم تعيينها عند اختيار نوع المهمة
```
