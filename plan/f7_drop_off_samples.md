## نظرة عامة

**DROP OFF SAMPLES** هو الخيار الأخير في شاشة اختيار نوع المهمة.
الهدف منه: **التسليم النهائي** للأكياس التي تم سحبها مسبقاً من حاويات السيارة. في هذه المرحلة، حالة المهام تكون `OUT_FREEZER` (أي خارج الثلاجة وجاهزة للتسليم للمختبر أو المستشفى).

**المنطق الأساسي للتدفق (حسب الكود الفعلي):**
1. **التجميع (Grouping):** يختار السائق "بطاقة وجهة" من قائمة المهام المفلترة على الحالة `OUT_FREEZER`.
2. **تخطي التحقق من الموقع:** برمجياً في كود الأندرويد الحالي، تم **تعطيل/تخطي** شاشتي التحقق من باركود الموقع وتوكن المهمة (`OutFreezerLocationCheckFragment` و `OutFreezerScanTaskTokenFragment`)، لينتقل التطبيق فوراً لشاشة مسح الأكياس للتسليم.
3. **مسح الأكياس محلياً:** تظهر قائمة بكل الأكياس المطلوبة للتسليم لهذه الوجهة. السائق يمسح الأكياس واحداً تلو الآخر لتأكيد استخراجها. **(تتم المطابقة محلياً فقط دون استدعاء API لكل مسحة)**.
4. **شاشة التوقيع:** بعد مسح جميع الأكياس، تظهر شاشة التوقيع لتأكيد الاستلام.
5. **الإغلاق النهائي:** يتم استدعاء الـ API الخاص بالإغلاق لإرسال بيانات المهام المجمعة وصورة التوقيع وتحديث الحالة إلى `DELIVERED`.

## Screen 1: OutFreezerTaskListFragment — شاشة قائمة مهام التسليم
### الكلاس الأصلي: `OutFreezerTaskListFragment.java` & `OutFreezerListTaskAdapter.java`

> **🚨 ملاحظة هامة جداً (لا توجد شاشة خريطة داخلية هنا!):** 
> على عكس مهام السحب (NEW) التي تفتح شاشة خريطة مخصصة (`ListMapFragment`)، فإن مهام التسليم (`OUT_FREEZER`) **لا تمتلك شاشة خريطة داخلية**. كل الإجراءات تتم مباشرة من نفس بطاقة المهمة (Card) في القائمة.

### 1. المنطق البرمجي لواجهة البطاقة (Card UI & Logic)
عند عرض المهام في القائمة، كل بطاقة تحتوي على الآتي:
1. **أيقونة الموقع (Location Icon):**
   - عند الضغط عليها، يتم الخروج من التطبيق وفتح **تطبيق خرائط جوجل الخارجي** مباشرة للتوجه لموقع العميل عبر الرابط: `https://www.google.com/maps/search/?api=1&query={lat},{lng}`.
2. **زر التحقق من الـ OTP (إن وجد):**
   - إذا كانت المهمة قادمة من نظام "عينتي" وتحتوي على OTP التسليم (`task.ayenati == true` ولديه `dropOffOtp`)، يظهر زر للتحقق من الـ OTP أولاً. لا يمكن إكمال التسليم قبل إدخاله ومطابقته.
3. **زر الوصول (Reach / Accept):**
   - بمجرد التحقق من الـ OTP (أو إذا لم يكن مطلوباً)، يظهر زر "الوصول".
   - عند الضغط عليه، يتم استدعاء API تأكيد الوصول `reachToLocationByDriver`.
   - عند نجاح الـ API، يختفي الزر وتصبح حالة المهمة (مؤكدة الوصول).
4. **الدخول للمهمة:**
   - السائق يضغط على البطاقة نفسها (Row Click).
   - لا يمكن الدخول إلا إذا كانت المهمة "مؤكدة الوصول".
   - عند الدخول يتم تجاوز شاشات (Location Check) و (Task Token) برمجياً والانتقال **فوراً** لشاشة مسح الأكياس (`OutFreezerScannedBagsFragment`).

```text
عند فتح الشاشة (getTaskList):
├── استدعاء API جلب قائمة المهام (getOutFreezerTasks) بحالة "OUT_FREEZER".
├── الخادم يرجع المهام مجمعة حسب الوجهة.
└── تفاعل السائق مع القائمة:
    ├── يضغط أيقونة الخريطة ← يفتح Google Maps.
    ├── يضغط زر الوصول ← يستدعي API الوصول.
    └── يضغط على البطاقة ← ينتقل لشاشة مسح الأكياس (Scanned Bags).
```

## الشاشات المتخطاة بالتفصيل (Details of Bypassed Screens)

### 1. شاشة التحقق من باركود الموقع — OutFreezerLocationCheckFragment
* **الكلاس الأصلي:** [OutFreezerLocationCheckFragment.java](file:///Users/ayaghoury/tracking_android/app/src/main/java/com/blazma/logistics/fragments/main/outfreezer/OutFreezerLocationCheckFragment.java)
* **الغرض:** التحقق من أن السائق متواجد بالفعل في موقع التسليم الصحيح عبر مسح باركود المكان المطبوع في المختبر أو العيادة.
* **التدفق والمنطق (Flow & Logic):**
  1. **تشغيل الكاميرا:** تفتح الشاشة وتبدأ الكاميرا بمسح الباركود تلقائياً بعد التحقق من صلاحية الكاميرا.
  2. **فلترة وتدقيق تنسيق الباركود:**
     * يتم استقبال القيمة النصية في الدالة `checkLocationID(scannedBarcode)`.
     * يتم فصل النص باستخدام رمز الشرطة `-` (دالة `split("-")`).
     * **الشرط:** يجب أن يتكون الباركود من جزأين فقط، والجزء الثاني يجب أن يكون كلمة `"location"` حرفياً.
     * **مثال صالح:** `45-location` (حيث `45` هو رقم الموقع).
     * إذا كان الباركود غير مطابق لهذا التنسيق، يتم الانتقال لدالة الخطأ `showFailedResultView()`.
  3. **تخزين المعرف مؤقتاً:** إذا نجح الشرط، يتم تخزين معرف الموقع كعدد صحيح:
     `UserInfo.getInstance().toLocationID = Integer.parseInt(strArray[0]);`
  4. **استدعاء API التحقق:** يتم استدعاء دالة `checkLocationInfo()` لإرسال طلب الشبكة.

* **مواصفات الـ API (API Specifications):**
  * **الرابط (Endpoint):** `POST` لـ `tasks/location/check`
  * **جسم الطلب (Request Body - JSON):**
    ```json
    {
      "tasks": [taskIds...], // قائمة معرفات المهام المحددة (مثال: [102, 103])
      "takasi_number": "",   // يتم إرسال نص فارغ في هذه الخطوة
      "to_location": 45      // رقم الموقع المستخرج من الباركود
    }
    ```
  * **التدفق بعد الاستجابة (Response Behavior):**
    * **في حال النجاح (`status: true`):** ينتقل التطبيق فوراً لشاشة مسح الأكياس للتسليم `OutFreezerScannedBagsFragment`.
    * *(ملاحظة: في الكود البرمجي الأصلي كان هناك زر Proceed يوجه إلى شاشة توكن المهمة `OutFreezerScanTaskTokenFragment` عبر دالة `goToNext` ولكن الكود الفعلي عند نجاح الـ API يتجاوزها ويذهب مباشرة لشاشة الأكياس).*
    * **في حال الفشل (`status: false`):** يتم استدعاء `showFailedResultView()` التي تقوم بـ:
      * إيقاف الكاميرا وعرض رسالة خطأ باللون الأحمر ("Invalid Location Bar Code").
      * تغيير نص زر المتابعة إلى "Rescan". عند نقر السائق عليه، يتم إخفاء التنبيه وإعادة تشغيل الكاميرا للمسح مجدداً.

### 2. شاشة مسح توكن المهمة — OutFreezerScanTaskTokenFragment
* **الكلاس الأصلي:** [OutFreezerScanTaskTokenFragment.java](file:///Users/ayaghoury/tracking_android/app/src/main/java/com/blazma/logistics/fragments/main/outfreezer/OutFreezerScanTaskTokenFragment.java)
* **الغرض:** إضافة خطوة تحقق ثنائية عبر مسح باركود يسمى (Takasi / Task Barcode) أو توكن المهمة المطبوع على أوراق الطلب.
* **التدفق والمنطق (Flow & Logic):**
  1. **مسح التوكن:** يقوم السائق بمسح أي باركود بالكاميرا.
  2. **تخزين القيمة:** يتم حفظ القيمة الممسوحة في متغير `scannedTaskToken = barcode.getRawValue()`.
  3. **زر التأكيد:** عند القراءة الناجحة، تظهر حالة النجاح ويتحول زر المتابعة إلى "Proceed".
  4. **استدعاء API التحقق:** عند ضغط "Proceed"، يتم استدعاء دالة `checkLocationInfo()` مع تمرير التوكن الممسوح.

* **مواصفات الـ API (API Specifications):**
  * **الرابط (Endpoint):** `POST` لـ `tasks/location/check` (نفس رابط التحقق من الموقع)
  * **جسم الطلب (Request Body - JSON):**
    ```json
    {
      "tasks": [taskIds...],          // قائمة معرفات المهام المحددة (مثال: [102, 103])
      "takasi_number": "TOKEN_12345", // قيمة التوكن الممسوحة من السائق
      "to_location": 45               // معرف الموقع المحفوظ من الخطوة السابقة
    }
    ```
  * **التدفق بعد الاستجابة (Response Behavior):**
    * **في حال النجاح (`status: true`):** ينتقل التطبيق إلى شاشة مسح الأكياس للتسليم `OutFreezerScannedBagsFragment` عبر دالة `goToNext()`.
    * **في حال الفشل (`status: false`):** يتم عرض رسالة الخطأ القادمة من السيرفر وتحويل الزر لـ "Rescan" لإعادة المحاولة.

## Screen 2: OutFreezerScannedBagsFragment — شاشة مسح الأكياس
### الكلاس الأصلي: `OutFreezerScannedBagsFragment.java`

هذه الشاشة تختلف جذرياً عن شاشة (Pull Out) في نقطتين أساسيتين: **لا يوجد استدعاء API لكل مسحة، ولا يوجد فصل حسب الحاويات.**

### 1. المنطق البرمجي لإعداد القائمة
```text
عند فتح الشاشة (initData):
├── يتم المرور على جميع المهام داخل البطاقة المجمعة المحددة.
├── يتم استخراج جميع الأكياس وتوحيدها في قائمة واحدة (dataList).
└── يتم إزالة أي تكرار محتمل (isExistedBagCode).
```

### 2. تدفق المسح (Local Scanning)
```text
عند مسح السائق لباركود كيس (doAction):
├── يبحث التطبيق هل الكيس الممسوح موجود ضمن القائمة المحلية؟
│   ├── إذا "نعم": 
│   │   └── يتم حذف الكيس فوراً من القائمة المعروضة.
│   └── إذا "لا": 
│       └── يظهر Toast خطأ (Barcode not matched).
```

### 3. تدفق الانتقال
```text
كلما تم مسح كيس، يتحقق التطبيق من (dataList.size() == 0):
├── إذا لم تفرغ: زر SAVE يظل مخفياً.
└── إذا فرغت بالكامل: يظهر زر SAVE وعند الضغط عليه ينتقل لـ `OutFreezerSignatureFragment`.
```

## Screen 3: OutFreezerSignatureFragment — شاشة التوقيع والإغلاق النهائي
### الكلاس الأصلي: `OutFreezerSignatureFragment.java`

### 1. عناصر الواجهة
- **اسم الوجهة:** يظهر في الأعلى (`txtLocationName`).
- **قائمة الأكياس:** تعرض مرة أخرى كملخص.
- **لوحة التوقيع:** `signatureView` لرسم التوقيع.
- **زر الحذف (Delete):** لمسح التوقيع الحالي ورسمه مجدداً.
- **زر Submit Task:** لإنهاء التدفق.

### 2. المنطق البرمجي للإنهاء (Submit Task)
عند الضغط على زر **Submit Task**:
1. يطلب التطبيق صلاحيات التخزين لحفظ صورة التوقيع محلياً (إذا استلزم الأمر).
2. يتم تحويل التوقيع المرسوم إلى صورة (Bitmap/File).
3. يتم استدعاء API الإغلاق المجمع `tasks/close` وإرسال:
   - المصفوفة المجمعة لـ `taskIds` التابعة لهذه الوجهة.
   - رمز التأكيد `deliver_confirmationCode` (إن وُجد).
   - (إذا طلب الـ Backend، تُرفق صورة التوقيع كـ Multipart).
4. عند نجاح الطلب، يتم نقل السائق إلى شاشة النجاح (TaskStatusFragment).

## مدخلات ومخرجات كافة الـ APIs بالتفصيل (API Specifications)

### 1. API: جلب مهام OUT_FREEZER للتسليم
* **الغرض:** جلب المهام الجاهزة للتسليم للعميل/المختبر.
* **رابط الاتصال (Endpoint):** `POST` لـ `driver/client/tasks`
* **المدخلات (Request Body):**
  ```json
  {
    "driver_id": 23,
    "status": "OUT_FREEZER"
  }
  ```
* **المخرجات الناجحة:** قائمة مهام تشمل تفاصيل المواقع و `taskIds` المجمعة.

### 2. API: الإغلاق النهائي المجمّع
* **الغرض:** إنهاء المهام بشكل نهائي وتحديث حالتها إلى `DELIVERED`.
* **رابط الاتصال (Endpoint):** `POST` لـ `tasks/close` (بصيغة `Multipart`)
* **المدخلات (Request Body):**
  - `@Part("tasks")`: المصفوفة المجمعة `[45, 46]` مرسلة كـ `RequestBody`.
  - `@Part("deliver_confirmationCode")`: كود التأكيد إن وجد.
  - (في حال طلب الخادم، يتم رفع صورة التوقيع كـ `@Part MultipartBody.Part`).
